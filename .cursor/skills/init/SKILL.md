# Init

## 목적
- 새 프로젝트/모듈 시작 시 반복되는 초기 작업을 일관된 순서로 수행한다.
- 기본값으로 안전한 스캐폴딩을 만들고, 누락되기 쉬운 문서/설정 체크를 보장한다.

## 사용 트리거
- 사용자가 `init`, `initialize`, `bootstrap`, `프로젝트 초기화`, `기본 세팅`을 요청할 때
- 새 기능 시작 전에 최소 구조/문서/설정을 먼저 준비하라고 할 때

## 실행 원칙
1. 먼저 현재 저장소 상태를 점검한다 (`git status`, 기존 파일 충돌 여부).
2. 이미 존재하는 파일은 덮어쓰기 전에 사용자 의도를 확인한다.
3. 초기화 작업은 작은 단위로 나눠 적용하고 각 단계마다 검증한다.
4. 초기화 후에는 변경 파일 목록과 다음 액션(테스트/커밋)을 제안한다.

## 기본 초기화 워크플로우
다음 체크리스트를 복사해 진행 상태를 관리한다.

```markdown
Init Progress:
- [ ] 1) 요구사항 확인 (스택, 타깃 플랫폼, 필수 산출물)
- [ ] 2) 기존 구조 점검 (충돌/중복 파일 확인)
- [ ] 3) 기본 디렉터리/파일 스캐폴딩
- [ ] 4) 설정 파일 템플릿 작성
- [ ] 5) README/개발 가이드 초안 작성
- [ ] 6) 린트/빌드/테스트 가능한 최소 검증
- [ ] 7) 결과 요약 및 후속 작업 제안
```

## 산출물 가이드(기본)
- 필수:
  - 프로젝트 루트 `README.md`
  - 실행/개발에 필요한 최소 설정 파일 템플릿
- 선택:
  - `docs/` 하위 초기 문서
  - `scripts/` 하위 초기화 보조 스크립트

## 출력 형식
초기화 완료 시 아래 형식으로 보고한다.

```markdown
## Init 결과
- 생성/수정 파일:
  - `path/to/file`
- 적용 이유:
  - 왜 이 파일/설정을 추가했는지 1줄 설명
- 검증:
  - 실행한 검증 명령과 결과
- 다음 단계:
  - 사용자가 바로 이어서 할 작업 1~3개
```

## 주의사항
- 비밀정보(`.env` 실값, 키/토큰)는 직접 생성하지 않고 예시 템플릿만 제공한다.
- 파괴적 명령(`reset --hard`, 강제 푸시 등)은 사용하지 않는다.
- 초기화 작업이 큰 경우, 먼저 작은 안전한 초안 구조를 만든 뒤 단계적으로 확장한다.
